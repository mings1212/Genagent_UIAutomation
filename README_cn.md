# 为什么要做这个项目？
 
我是一个UI自动化工程师，我深知在许多公司里，UI自动化的普及性远不如API自动化，许多公司甚至完全不考虑UI自动化，因为投入产出比不高。
</br>
在过去的6个月里，我积极尝试有AI加持的UI自动化框架，比如Midscene，Browser-use等，不得不说它们相比于传统的UI自动化框架了质的进步，但初步的惊喜过后，我发现我仍然要写许多代码来管理用例，配置包名，驱动设备，特别是iOS17+，这简直是噩梦，我相信有人懂我在讲什么🥲
</br>
后来，在测开团队为测试团队做“难点攻克”的工作时，我意识到可以把多个解决方案集成到一个框架里，让测试人员不用为测试以外的事情分散精力，让专业的人做专业的事，于是就有了这个项目。
</br>
# 为什么说它是不一样的？
用一句话总结就是：它不需要测试人员关注测试以外的事情。
</br>
这个世界上已经有太多轮子了，各种框架层出不穷，我们来学习研究它们并封装好，让测试人员专注于业务方面的问题，谁能更深刻的理解业务，谁就是公司里最重要的人。
</br>
以下是该项目具备的几个特征：
## 0成本部署
系统已集成安卓/iOS/鸿蒙/web等驱动程序并封装连接设备、打开目标APP/URL、驱动设备执行等UI自动化必要流程，拿到该系统即可书写/执行用例，完全无需考虑冗长复杂的UI自动化工程启动流程。
<img width="100%" src="./images/图片1.png" alt="" />
## 0脚本
系统集成了脑图用例管理系统，用户使用自然语言描述用例流程与断言即可，系统使用大模型能力理解用户意图、执行相关元素、给出断言结果，相比传统的UI自动化工程，case书写效率有质的飞跃，且易于审查与调整。
<img width="100%" src="./images/图片2.png" alt="" />
## 超强跨端
本系统可以在window、mac、linux三大主流PC/服务器系统上驱动安卓、iOS、鸿蒙、web执行用例并给出一致的结果，用户完全无需考虑适配平台所面临的复杂工程问题。系统借助大模型能力消弭了平台差异并做出优化调整，做到了一份用例，多处执行。
